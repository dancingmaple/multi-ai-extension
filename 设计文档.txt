

# 一、项目目标

开发一个浏览器插件，实现以下能力：

1. 用户在插件侧边栏输入一次问题
2. 插件将问题**并发发送**到多个已打开或自动打开的 AI 网页：
   - ChatGPT
   - Gemini
   - DeepSeek
3. 插件通过网页自动化：
   - 找到输入框
   - 填入 prompt
   - 点击发送
   - 监听页面回答流式更新
4. 插件统一收集多个 AI 的回答
5. 用户可在插件侧边栏中通过 **tab** 查看每个 AI 的回答
6. 支持基础状态展示：
   - waiting
   - sending
   - streaming
   - done
   - error
7. 首个版本只做 **单轮问答**，暂不要求多轮上下文同步

---

# 二、实现边界

## 这版要做
- Chrome/Edge 浏览器插件
- Manifest V3
- Side Panel 作为主 UI
- 支持 3 个站点：
  - `chatgpt.com`
  - `gemini.google.com`
  - `chat.deepseek.com` 或当前可访问 DeepSeek 官方聊天网页
- 并发发送 prompt
- 流式采集回答文本
- 在插件中切换 tab 查看结果
- 自动打开目标页面
- 检测未登录/页面不可用状态
- 基础错误提示
- 本地保存最近一次任务状态

## 这版不做
- 多轮会话历史同步
- 富文本/代码块高保真解析
- 图片上传
- 文件上传
- 复杂反爬绕过
- 云端同步
- 账号管理
- 官方 API 集成

# 三、总体架构

采用 **插件分层架构**：

```text
Side Panel UI
   ↓
Background Service Worker（任务调度中心）
   ↓
Content Script（注入各 AI 网页）
   ↓
Site Adapter（站点自动化逻辑）
```

---

# 四、模块职责

## 1. Side Panel UI
职责：
- 输入 prompt
- 选择目标平台
- 发起任务
- 实时展示每个平台状态
- tab 切换不同 AI 的回答
- 展示错误信息
- 提供“重试”按钮

建议技术：
- React + TypeScript + Vite
- Zustand 做状态管理
- Tailwind 可选

---

## 2. Background Service Worker
职责：
- 接收 UI 发来的“群发任务”
- 检查目标站点标签页是否存在
- 不存在则创建
- 确保 content script 就绪
- 向各站点分发任务
- 汇总各站点流式消息
- 将结果广播给 side panel

它是整个系统的**总调度器**。

---

## 3. Content Script
职责：
- 在目标网页中执行自动化动作
- 调用对应站点适配器
- 向 background 回传：
  - 状态变化
  - 流式文本
  - 完成事件
  - 错误事件

---

## 4. Site Adapter
每个站点一个适配器，封装网页自动化细节。

接口统一为：

- `isReady()`
- `findInput()`
- `setPrompt()`
- `submit()`
- `startStreamCapture()`
- `detectDone()`
- `detectLoginRequired()`

支持站点：
- `chatgpt`
- `gemini`
- `deepseek`

---

# 五、核心流程

## 用户发起任务流程

```text
1. 用户在 Side Panel 输入 prompt
2. UI -> Background: ASK_ALL
3. Background 为每个平台：
   a. 查找已有 tab
   b. 若无则创建 tab
   c. 等待页面完成加载
   d. 向目标 tab 发送 EXECUTE_PROMPT
4. Content Script 收到消息后调用对应 adapter
5. Adapter:
   a. 检测页面是否可用/是否登录
   b. 找输入框
   c. 写入 prompt
   d. 点击发送
   e. 监听回答区域 DOM 变化
6. Content Script 持续发送 STREAM_UPDATE 给 Background
7. Background 转发给 Side Panel
8. Side Panel 更新对应 tab 内容
9. 当 adapter 判断完成后，发送 TASK_DONE
10. UI 状态更新为 done
```

# 十一、自动化实现细节

## 1. 输入文本的方法

很多网页是 React 控件，不能只写：

```ts
element.value = prompt;
```

必须用原生 setter + 事件派发。

```ts
export function setNativeInputValue(
  element: HTMLTextAreaElement | HTMLInputElement,
  value: string
) {
  const prototype = Object.getPrototypeOf(element);
  const descriptor = Object.getOwnPropertyDescriptor(prototype, 'value');
  descriptor?.set?.call(element, value);
  element.dispatchEvent(new Event('input', { bubbles: true }));
  element.dispatchEvent(new Event('change', { bubbles: true }));
}
```

如果是 contenteditable，则需要：

```ts
export function setContentEditableValue(element: HTMLElement, value: string) {
  element.focus();
  document.execCommand('selectAll', false);
  document.execCommand('insertText', false, value);
  element.dispatchEvent(new InputEvent('input', { bubbles: true, data: value }));
}
```

如 `execCommand` 不稳定，则降级：
- `textContent = value`
- 手动触发 input

---

## 2. 等待元素出现

```ts
export async function waitForElement(
  selectors: string[],
  timeoutMs = 15000
): Promise<HTMLElement> { ... }
```

实现方式：
- 先同步 query
- 不存在则用 `MutationObserver`
- 超时抛错

---

## 3. 回答流式监听

对回答容器使用 `MutationObserver`。

实现要求：
- 每 300ms 节流一次上报
- 只上报最新完整文本
- 去除重复连续内容
- 监听结束后 disconnect

---

## 4. 完成判断策略

建议 coding agent 实现一个通用 `DoneDetector`，按以下优先级判断：

### 信号 A：页面停止变化
- 回答容器文本在 150ms 内无变化

### 信号 B：停止按钮状态变化
- 如果“停止生成”按钮消失，则判定结束

### 信号 C：生成相关 DOM class 状态切换
- 若站点有明显状态 class，则作为补充

最终规则：
- 满足 A 且 B，或 A 单独持续 250ms，则视为完成

---

## 5. 未登录检测

每个 adapter 提供 `detectLoginRequired()`，检查：
- 登录按钮出现
- 页面被重定向到登录页
- 输入框不存在且有登录提示文案

状态统一上报为：
- `login_required`

# 十二、Background 调度设计

## 1. tabManager

职责：
- 查询是否已有目标站点 tab
- 若无则创建
- 若存在则返回最近一个
- 可选是否激活 tab，默认不激活

接口建议：

```ts
async function getOrCreateProviderTab(provider: ProviderName): Promise<number>
```

规则：
- 匹配目标域名
- 找不到就创建新 tab
- 等待 `status=complete`
- 额外等待 2~4 秒让 SPA 渲染稳定

---

## 2. taskManager

职责：
- 创建任务
- 追踪各 provider 状态
- 接收流式更新
- 存入内存 + chrome.storage.local

接口建议：

```ts
createTask(prompt: string, targets: ProviderName[]): AskTaskState
updateProviderStatus(...)
updateProviderContent(...)
finishProviderTask(...)
failProviderTask(...)
getTask(taskId: string)
```

---

## 3. messageRouter

职责：
- 统一处理来自 UI、content script 的消息
- 做消息校验
- 转发事件

---
